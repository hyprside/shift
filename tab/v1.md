# Tab Protocol (Tab v1)

Tab is the bidirectional protocol between client compositors and Shift for authentication, DMA-BUF framebuffer linking, per-frame fence commits, receiving routed input events, and driving session orchestration such as admin-controlled switches. This document walks through the full lifecycle—from the initial socket connection to high-FPS multi-monitor rendering—so implementors can confidently integrate with Shift using Tab v1.

## Tab Messages

All Tab messages are framed as **two UTF-8 lines** on a persistent Unix domain socket connection:

1. **Header line** – ASCII identifier naming the event type (e.g. `hello`, `frame_commit`).
2. **Payload line** – either a single-line JSON object, a raw string, or (for payload-less messages) a sentinel.

Lines are terminated with new line characters (`\n`) and there are no blank separators. JSON payloads describe most control plane messages, while `frame_commit` and `frame_done` use raw strings to minimize parsing overhead on the hot path. Messages without any payload use the four-byte sentinel `\0\0\0\0` for the second line to keep framing consistent. File descriptors (FDs) ride along the same `sendmsg` call via `SCM_RIGHTS`; Tab v1 never references FD indices inside JSON, so each header defines how many FDs (if any) must accompany it.


Example JSON message:

```text
auth
{"token":"adm_xVJbVARh0/XFVYJBZiRQWhotDWo="}
```

Raw-string frame commit:

```text
frame_commit
mon_aNwVCYV9h1EFs4tON1jgtca3ycY=
```

## Connections

Every compositor opens a single `AF_UNIX` `SOCK_STREAM` connection to `/tmp/shift.sock` and keeps it alive for the compositor’s lifetime. The socket carries both directions of traffic—Shift may send messages at any time (monitor hotplug, input, errors) and the compositor pushes auth, framebuffer links, and frame commits whenever needed.

### Connection Lifecycle

1. Connect to `/tmp/shift.sock`.
2. Receive `hello` from Shift, describing server identity and supported protocol version.
3. Send `auth` with the compositor’s token (admin or session scoped).
4. Receive either `auth_ok` or `auth_error`.
5. Admin clients (e.g. the login shell) may request new sessions via `session_create` and receive freshly minted tokens with `session_created`.
6. After `auth_ok`, record the announced session metadata and monitor IDs. Sessions are marked `loading` until they explicitly report ready.
7. Session-role clients must send `session_ready` once their desktop environment has finished booting; Shift will then broadcast the `session_state` update so admins know it is safe to switch.
8. For every monitor that should display this compositor, send `framebuffer_link` with a DMA-BUF FD describing the render target.
9. Enter the frame loop per monitor: render into the linked buffer, then send `frame_commit` with a fence FD; optionally consume `frame_done` acknowledgements.
10. Handle asynchronous events such as `input_event`, `monitor_added`, `monitor_removed`, `session_state`, and `session_active` while adjusting rendering accordingly.
11. On disconnect or fatal error, release GPU resources, close the socket, and reconnect if appropriate.

## Hello (hello)

**Direction:** Shift → Client
**Header:** `hello`
**Payload:** JSON object

Shift always initiates a fresh connection by advertising its identity and supported protocol string. This lets the compositor verify that it truly reached Shift (and not some stale stub) and that both ends agree on the framing rules before exchanging sensitive data.

```ts
type HelloPayload = {
	server: string;   // Human-readable instance string such as "Shift v0.1.0-alpha"
	protocol: string; // SemVer-scoped identifier like "tab/v1.0.0"
};
```

Clients must verify that `protocol` matches the Tab version they implement and close the connection if unsupported. Shift encodes the crate version inside the string, so a future `tab/v1.1.0` will still share framing but may add optional fields; incompatible breaking releases will increment the major portion (`tab/v2.0.0`).

```text
hello
{"server":"Shift v0.1.0-alpha","protocol":"tab/v1.0.0"}
```

## Authentication (auth, auth_ok, auth_error)

### auth

**Direction:** Client → Shift
**Header:** `auth`
**Payload:** JSON object

Immediately after `hello`, the compositor proves which session it belongs to (or that it is the admin shell) by sending a single token. Without this step Shift would not know which session graph to expose or whether the caller is authorized to trigger global actions.

```ts
type AuthPayload = {
	token: string; // Session or admin token
};
```

Session tokens bind the compositor to a specific session, while admin tokens unlock session switching and management capabilities.

```text
auth
{"token":"ses_t2YDzCRkLK7Pt9gVTvIv4w=="}
```

### auth_ok

**Direction:** Shift → Client
**Header:** `auth_ok`
**Payload:** JSON object

An authenticated compositor receives a snapshot of the session it is bound to so that it can immediately size buffers correctly and know whether it should currently present frames. Shift sends this once per successful login and again only if the connection is re-established.

```ts
type AuthOkPayload = {
	session: SessionInfo;
};

type SessionInfo = {
	id: string;
	role: SessionRole;
	display_name?: string | null;
	state: SessionLifecycle;
	is_active: boolean;
	cursor_position: [number, number];
	monitors: MonitorInfo[];
};

type SessionRole = 'admin' | 'session';
type SessionLifecycle = 'pending' | 'loading' | 'occupied' | 'consumed';

type MonitorInfo = {
	id: string;
	width: number;
	height: number;
	refresh_rate: number;
	name: string;
};
```

Clients must cache monitor IDs for later `framebuffer_link` and `frame_commit` messages.

```text
auth_ok
{"session":{"id":"se_rjJO4LcZuOPVCdojzs9ea3ohkv8=","role":"admin","display_name":null,"state":"occupied","is_active":true,"cursor_position":[960,540],"monitors":[{"id":"mon_aNwVCYV9h1EFs4tON1jgtca3ycY=","width":1920,"height":1080,"refresh_rate":60,"name":"eDP-1"}]}}
```

### auth_error

**Direction:** Shift → Client
**Header:** `auth_error`
**Payload:** JSON object

Whenever Shift rejects the provided token (unknown, revoked, or already bound elsewhere) it responds with `auth_error` and shuts the socket down to avoid any unauthenticated traffic.

```ts
type AuthErrorPayload = {
	error: string; // Human-readable explanation
};
```

Shift closes the socket immediately after sending `auth_error`.

```text
auth_error
{"error":"invalid_token"}
```

## Framebuffer Linking (framebuffer_link)

**Direction:** Client → Shift
**Header:** `framebuffer_link`
**Payload:** JSON object
**FDs:** Exactly 1 DMA-BUF FD (SCM_RIGHTS)

This message establishes the zero-copy path per monitor: after authentication but before committing frames, the compositor hands Shift a DMA-BUF to import for scan-out. Shift keeps this association until the compositor resends `framebuffer_link` for that monitor.

```ts
type FramebufferLinkPayload = {
	monitor_id: string;
	width: number;
	height: number;
	stride: number;
	offset: number;
	fourcc: number; // DRM fourcc as int (e.g. 875708993 for ARGB8888)
};
```

Shift imports the DMA-BUF and associates it with the monitor. Resend `framebuffer_link` when reallocating the buffer, changing format/resolution, or when instructed by Shift.

```text
framebuffer_link
{"monitor_id":"mon_aNwVCYV9h1EFs4tON1jgtca3ycY=","width":1920,"height":1080,"stride":7680,"offset":0,"fourcc":875708993}
```

## Frame Loop (frame_commit, frame_done)

Tab’s hot path focuses on fence passing, so payloads are lean and uniquely identify the monitor.

### frame_commit

**Direction:** Client → Shift
**Header:** `frame_commit`
**Payload:** Raw string monitor ID
**FDs:** Exactly 1 sync fence FD

Each time a monitor-sized render completes, the compositor notifies Shift via `frame_commit`. This gives Shift a synchronization primitive so it can wait for the GPU to finish before flipping the buffer onto the targeted monitor, ensuring glitch-free output without polling.

```ts
type FrameCommitPayload = string; // monitor_id
```

Send once per presented frame per monitor after finishing GPU rendering into the linked DMA-BUF. Shift waits on the provided fence before scanning the buffer out.

```text
frame_commit
mon_aNwVCYV9h1EFs4tON1jgtca3ycY=
```

### frame_done

**Direction:** Shift → Client
**Header:** `frame_done`
**Payload:** Raw string monitor ID
**FDs:** None

When enabled, Shift mirrors the display pipeline status back to the compositor. This optional acknowledgement communicates that a frame for the referenced monitor has been latched or displayed, providing feedback loops for throttling or advanced pacing heuristics.

Sent once per frame per monitor in implementations that provide back-pressure feedback. Clients may use this to throttle frame production or gather latency metrics but must remain compatible with deployments that never send `frame_done`.

```ts
type FrameDonePayload = string; // monitor_id
```

```text
frame_done
mon_aNwVCYV9h1EFs4tON1jgtca3ycY=
```

## Monitor Management (monitor_added, monitor_removed)

### monitor_added

**Direction:** Shift → Client
**Header:** `monitor_added`
**Payload:** JSON object

Monitors can be hotplugged at runtime. Whenever Shift gains a new display routed to the current session, it pushes `monitor_added` so the compositor can advertise it in UI and allocate a matching framebuffer.

```ts
type MonitorAddedPayload = {
	monitor: MonitorInfo;
};
```

Upon receipt, clients may immediately issue `framebuffer_link` for the new monitor when ready.

```text
monitor_added
{"monitor":{"id":"mon_ext0","width":2560,"height":1440,"refresh_rate":144,"name":"DP-1"}}
```

### monitor_removed

**Direction:** Shift → Client
**Header:** `monitor_removed`
**Payload:** JSON object

Similarly, when a monitor leaves the topology (physical unplug, admin reassign, etc.), Shift notifies clients so they can tear down buffers and halt frame commits targeting a nonexistent display.

```ts
type MonitorRemovedPayload = {
	monitor_id: string;
	name: string;
};
```

Clients must stop sending `frame_commit` for the removed monitor and release corresponding resources.

```text
monitor_removed
{"monitor_id":"mon_ext0","name":"DP-1"}
```

## Input Events (input_event)

Shift only forwards input to whichever session is active. Whenever Shift receives hardware events destined for this compositor, it emits `input_event` so the compositor can update pointer focus, keyboard state, gesture recognizers, or tablet interactions. Each event mirrors a libinput event 1:1 (same semantic fields and timestamps) and is distinguished via the `kind` discriminator.

**Direction:** Shift → Client
**Header:** `input_event`
**Payload:** JSON object (discriminated by `kind`)
**FDs:** None

	| PointerMotionAbsoluteEvent
	| PointerButtonEvent
	| PointerAxisEvent
	| PointerAxisStopEvent
	| PointerAxisDiscreteEvent

## Session Management (session_create, session_created)

Admin clients orchestrate session lifecycles: they solicit new desktops, hand the resulting tokens to per-user compositors, and tear them down when a user logs out. Tab adds lightweight create/ack messages so that the login shell never needs to mint tokens out-of-band.

### session_create

**Direction:** Admin Client → Shift
**Header:** `session_create`
**Payload:** JSON object
**FDs:** None

Requests that Shift create a new inactive session placeholder. Shift will reserve internal resources, mint a single-use token, and emit `session_created` in response. Admin clients typically send one `session_create` per user login attempt (e.g. when launching a desktop compositor after credentials are verified).

```ts
type SessionCreatePayload = {
	role: SessionRole;
	display_name?: string | null; // Optional UI label for admin surfaces
};
```

```text
session_create
{"role":"session","display_name":"alice"}
```

### session_created

**Direction:** Shift → Admin Client
**Header:** `session_created`
**Payload:** JSON object
**FDs:** None

Shift answers each `session_create` with the full session descriptor plus a token that the newly spawned compositor must use in its `auth` message. Until a compositor connects with that token, the session remains inactive and has no monitors assigned.

```ts
type SessionCreatedPayload = {
	session: SessionInfo;
	token: string; // Pass to the compositor that will bind this session
};
```

```text
session_created
{"session":{"id":"se_user2","role":"session","is_active":false,"cursor_position":[0,0],"monitors":[]},"token":"ses_QneY60mnZsYt2c9RgK+1CA=="}
```

### session_ready

**Direction:** Session Client → Shift
**Header:** `session_ready`
**Payload:** JSON object
**FDs:** None

Newly launched desktops often need to start XDG portals, DBus services, and compositor shells before they can accept focus. Until they finish bootstrapping, the session remains in the `loading` state and Shift should not switch to it. Once initialization completes, the session client must emit `session_ready` so Shift can mark it `occupied` and alert admin clients.

```ts
type SessionReadyPayload = {
	session_id: string;
};
```

Shift verifies that the `session_id` matches the authenticated token and that the session was previously `loading`. After a successful update, it broadcasts `session_state` to every admin client.

```text
session_ready
{"session_id":"se_user2"}
```

### session_state

**Direction:** Shift → Admin Client
**Header:** `session_state`
**Payload:** JSON object
**FDs:** None

Whenever a session advances through the lifecycle (`pending` → `loading` → `occupied` → `consumed`), Shift notifies all connected admin clients so they can drive UI updates, enable transitions, or prune dead entries.

```ts
type SessionStatePayload = {
	session: SessionInfo;
};
```

Admin clients should treat `pending` as “token minted but no compositor connected”, `loading` as “compositor connected but not ready”, `occupied` as “ready to display/switch to”, and `consumed` as final (the token may not be reused).

```text
session_state
{"session":{"id":"se_user2","role":"session","display_name":"alice","state":"occupied","is_active":false,"cursor_position":[0,0],"monitors":[]}}
```

## Session Control (session_switch, session_active)

### session_switch

**Direction:** Admin Client → Shift
**Header:** `session_switch`
**Payload:** JSON object
**FDs:** None

Only the admin compositor may tell Shift to foreground a different session. `session_switch` encapsulates that request along with an optional transition hint so Shift can orchestrate animations while reassigning monitors and input focus.

```ts
type SessionSwitchPayload = {
	session_id: string;
	animation: string | null; // e.g. "fade", "slide_right", etc.
};
```

Animations convey intent only; Shift may downgrade unknown values to a default effect or return an `error`.

```text
session_switch
{"session_id":"se_user2","animation":"slide_left"}
```

### session_active

**Direction:** Shift → Client
**Header:** `session_active`
**Payload:** JSON object
**FDs:** None

Whenever Shift moves input focus between sessions—because of `session_switch`, policy changes, or power-state events—it informs all connected compositors. The active compositor can keep animating, while inactive ones may pause heavy GPU work or dim their UI.

```ts
type SessionActivePayload = {
	session_id: string;
};
```

Non-active compositors may reduce rendering load or pause entirely upon receiving this event.

```text
session_active
{"session_id":"se_user2"}
```

## Keepalive (ping, pong)

`ping`/`pong` messages are optional but useful for detecting dead connections or measuring latency without touching the critical frame loop. Both frames omit payloads (the second line is the sentinel `\0\0\0\0`), so only the header string matters.

### ping

**Direction:** Either
**Header:** `ping`
**Payload:** None (`\0\0\0\0`)
**FDs:** None

Compositors or Shift can opportunistically send `ping` to ensure the socket remains responsive when no other traffic flows—particularly handy for idle sessions that might otherwise only exchange frames sporadically.

### pong

**Direction:** Either (usually in direct reply to `ping`)
**Header:** `pong`
**Payload:** None (`\0\0\0\0`)
**FDs:** None

Many implementations still track round-trip times by timestamping when the header is sent/received; if correlation IDs are needed, an out-of-band mechanism (e.g. alternating ping/pong cadence per device) must be used because the current wire format does not carry programmable payload data.

## Errors (error, auth_error)

### error

**Direction:** Shift → Client
**Header:** `error`
**Payload:** JSON object
**FDs:** None

Shift emits `error` whenever it needs to report a recoverable problem while keeping the connection alive—such as a malformed message or a bad DMA-BUF. Logging these codes makes it easier to diagnose compositor integration issues without digging through kernel traces.

```ts
type ErrorPayload = {
	code: string;
	message: string | null;
};
```

Example payload:

```text
error
{"code":"unknown_monitor","message":"No monitor named mon_ext0 in this session"}
```

Representative `code` values:

- `unknown_monitor` – monitor ID not recognized for this session.
- `invalid_texture_fd` – DMA-BUF import or validation failed.
- `protocol_violation` – wrong FD count, malformed payload, or unexpected header.
- `unsupported_version` – client attempted to speak an older/newer Tab revision.

Some errors are recoverable (e.g. retry `framebuffer_link`), while others prompt Shift to close the socket; clients should treat closures as fatal and reconnect cleanly.

### auth_error reminder

`auth_error` appears only during authentication and is always followed by a disconnect, so the client must halt initialization and alert the user or supervisor process before attempting a fresh connection. Treat repeated `auth_error` responses as a provisioning issue rather than a transient transport failure.
